# arumono-recipe 要件定義書

## 1. はじめに

### 1.1 文書の目的

本文書は、家庭の在庫食材から献立を提案するWebアプリ「arumono-recipe」の要件を定義し、開発スコープと品質基準について関係者間で合意することを目的とする。

### 1.2 対象読者

- 本アプリの開発者
- 本アプリのオーナー兼利用者（家族メンバー）

### 1.3 用語定義

- **世帯**: 食材在庫を共有する家族グループ。1ユーザーは複数世帯に所属できる
- **在庫食材**: 世帯ごとに登録された、現時点で家にある食材の集合
- **常備食材**: 醤油・塩・スパイスなど、日常的にストックしている食材・調味料
- **レシピ提案**: 在庫食材を元に複数件のレシピ候補をユーザーに提示する機能
- **お気に入り**: ユーザーが再利用したい意思を示してマークしたレシピ
- **人前**: レシピを何人分作るかを示す数値。在庫消費量のスケーリングに用いる

## 2. システム概要

### 2.1 背景・課題

- 冷蔵庫にある食材を把握しきれず、食品ロスが発生している
- 日々の献立を考える意思決定コストが高い
- 家族間で在庫情報を共有しづらく、重複買いや使い忘れが起きる

### 2.2 目的・ゴール

在庫食材を起点とした献立提案・調理実行・在庫消費を1つのアプリ内で完結させ、食品ロスの削減と献立決定の負担軽減を実現する。

### 2.3 ターゲットユーザー

- 共働き世帯
- 自炊を日常的に行う家庭
- 家族で食材を共有・管理したいユーザー

### 2.4 提供価値

- 「あるもの」起点でレシピが提案され、献立決定が早い
- 提案→調理→在庫消費が1サイクルで自動連動し、台帳が自然に更新される
- 家族間で在庫とレシピ履歴を共有できる

### 2.5 システム全体イメージ

```text
[ クライアント (PWA / スマホブラウザ) ]
            |
            v
[ アプリケーションサーバ / API ]
            |
   +--------+---------+----------------+
   v                  v                v
[ DB ]         [ 外部AIサービス ]  [ メール送信 ]
 食材           - LLM (提案/補正)    招待・通知
 レシピ          - 画像生成
 お気に入り      - OCR (レシート)
 世帯/ユーザー
```

## 3. 業務要件 / ユースケース

主要ユースケースを以下に示す。詳細シナリオは設計フェーズで補足する。

- **UC-01 世帯作成と招待**: 新規ユーザーが世帯を作成し、招待コードを発行して家族メンバーを追加する
- **UC-02 レシート画像による食材一括登録**: ユーザーが買い物のレシートを撮影しアップロードすると、OCRとAI補正を経て食材候補が抽出され、ユーザーが内容を確認・修正して在庫に登録する
- **UC-03 食材の手動追加・編集・削除**: ユーザーが食材名・数量・単位・賞味期限（任意）を入力して在庫を更新する
- **UC-04 在庫食材からの献立提案**: ユーザーが提案条件（人数・調理時間・ジャンル等）を指定し、在庫を元にした複数件のレシピ提案を受け取る
- **UC-05 レシピ実行と在庫消費**: ユーザーが提案レシピを「作った」として確定すると、人前スケーリング後の使用量を在庫から自動減算する
- **UC-06 お気に入り登録**: ユーザーがレシピをお気に入り登録/解除し、一覧から再アクセスする
- **UC-07 個人最適化設定**: 世帯メンバーごとにアレルギー食材・苦手食材・好みのテイストを登録し、提案結果に反映させる

## 4. 機能要件

機能IDは `FR-xx` で表記し、優先度は MoSCoW（Must / Should / Could / Won't）で示す。

### 4.1 認証・世帯管理

- **FR-01 アカウント登録/ログイン** [Must]
  - メールアドレス＋パスワード、またはGoogle等のOAuthでログイン可能
  - パスワード再設定のためのメール送信
- **FR-02 世帯の作成・招待コード発行・参加** [Must]
  - ユーザーは世帯を新規作成できる
  - 世帯オーナーは招待コードを発行し、他ユーザーは招待コードで世帯へ参加できる
- **FR-03 世帯メンバー管理** [Should]
  - 世帯メンバー一覧の表示
  - メンバーの脱退、オーナーによる除名
  - メンバー権限（オーナー / 一般）の管理

### 4.2 食材在庫管理

- **FR-10 レシート画像アップロードによる食材一括登録** [Must]
  - レシート画像をアップロードするとOCR+AI補正により食材候補リストが生成される
  - 登録前に確認画面で食材名・数量・単位を編集可能
  - 確定後、在庫に一括追加する
- **FR-11 手入力での食材追加** [Must]
  - 食材名・数量・単位・賞味期限（任意）・カテゴリ・常備フラグを入力して登録する
- **FR-12 食材の編集・削除** [Must]
  - 既存の在庫食材の数量・賞味期限・カテゴリを編集できる
  - 不要な食材を削除できる
- **FR-13 食材一覧の表示** [Must]
  - カテゴリ別・賞味期限が近い順などでソート可能
  - 賞味期限が近い食材を視覚的に強調する
- **FR-14 調味料・常備食材の在庫管理** [Should]
  - 常備食材フラグで通常食材と区別する
  - レシピ提案時に不足している調味料を警告する

### 4.3 レシピ提案・閲覧

- **FR-20 在庫食材を元にしたレシピ複数件提案** [Must]
  - 既存レシピDBの検索結果と、LLMによる生成レシピをハイブリッドで提示する
  - 1回の提案で複数件（目安3〜5件）を返す
- **FR-21 提案レシピの詳細表示** [Must]
  - 材料・分量・手順・調理時間（分）・使用する在庫食材のハイライト
- **FR-22 完成イメージ画像の表示** [Must]
  - 既存画像があれば優先表示
  - 既存画像が無い場合はAI生成画像を表示する
- **FR-23 何人前を作るか選択** [Must]
  - 人前数を変更すると材料分量が自動でスケーリングされる
- **FR-24 提案条件の指定** [Should]
  - 調理時間上限（例: 30分以内）
  - ジャンル（和食 / 洋食 / 中華 / その他）
  - テイスト（さっぱり / こってり / ヘルシー など）
- **FR-25 提案レシピのDB蓄積** [Must]
  - 提案されたレシピは全てDBに保存し、再表示・再利用できるようにする

### 4.4 レシピ実行（在庫消費）

- **FR-30 「作った」操作による在庫減算** [Must]
  - 人前スケーリング後の使用量で、在庫食材を減算する
  - 在庫を超える減算が発生する場合は0で打ち止めとする
- **FR-31 減算前の確認ダイアログ** [Must]
  - 減算対象の食材と数量を一覧表示し、不足食材があれば警告する
  - ユーザーが確定操作を行うまで在庫を変更しない
- **FR-32 調理履歴の記録** [Should]
  - 誰がいつ何のレシピを何人前作ったかをログとして保持する

### 4.5 お気に入り

- **FR-40 レシピのお気に入り登録/解除** [Must]
  - レシピ詳細または一覧から1タップで操作できる
- **FR-41 お気に入り一覧の閲覧** [Must]
  - お気に入りレシピをジャンル・調理時間で絞り込める

### 4.6 個人最適化

- **FR-50 アレルギー食材の登録** [Must]
  - 世帯メンバーごとにアレルギー食材を登録する
- **FR-51 苦手食材の登録** [Should]
  - 世帯メンバーごとに苦手食材を登録する
- **FR-52 アレルギー食材の自動除外** [Must]
  - 提案時に、世帯メンバーいずれかのアレルギー食材を含むレシピを除外する
  - 苦手食材は降順優先度を下げる（任意の挙動）

## 5. 非機能要件

### 5.1 性能

- レシピ提案APIの応答時間: P95で10秒以内（LLM呼び出しを含む）
- 一般画面の表示時間: P95で2秒以内
- 画像生成は非同期処理を許容する（プレースホルダ表示）

### 5.2 可用性

- 月間稼働率: 99%以上（個人利用想定のため上限緩め）
- 計画停止は事前告知を行う

### 5.3 セキュリティ

- 通信はすべてHTTPSで暗号化する
- パスワードはbcrypt等のハッシュ関数で保存する
- 世帯データは所属世帯メンバー以外からはアクセスできない（行レベルアクセス制御）
- レシート画像はOCR完了後に速やかに削除する、または保存期間を限定する
- APIキー等の機密情報は環境変数で管理し、リポジトリにコミットしない

### 5.4 ユーザビリティ

- スマートフォン片手操作で、食材登録・レシピ確認・在庫消費の主要操作が完結する
- PWAとしてホーム画面に追加可能
- 主要操作は3タップ以内を目指す

### 5.5 保守性

- フロントエンドとバックエンドを分離した構成
- 外部AIサービス（LLM / OCR / 画像生成）はアダプタ層で抽象化し、差し替え可能とする
- 主要ロジックには単体テストを整備する

### 5.6 拡張性

- LLM・OCR・画像生成プロバイダを設定で差し替え可能
- データモデルは将来の機能拡張（買い物リスト、献立カレンダー等）を見据えて設計する

## 7. データ要件

主要エンティティと関連を以下に示す。物理スキーマは設計フェーズで確定する。

- **User**: `id`, `email`, `name`, `hashed_password`, `created_at`
- **Household**: `id`, `name`, `invite_code`, `created_at`
- **HouseholdMember**: `household_id`, `user_id`, `role`, `allergies[]`, `dislikes[]`, `taste_preferences[]`
- **Ingredient**: `id`, `household_id`, `name`, `quantity`, `unit`, `expires_at` (nullable), `is_pantry_staple`, `category`, `created_at`, `updated_at`
- **Recipe**: `id`, `household_id` (nullable: 共通レシピは null), `title`, `description`, `cuisine`, `taste`, `servings_default`, `cooking_time_min`, `image_url`, `source` (`ai` / `external` / `user`), `created_at`
- **RecipeIngredient**: `recipe_id`, `ingredient_name`, `quantity`, `unit`
- **RecipeStep**: `recipe_id`, `step_no`, `instruction`
- **Favorite**: `household_id`, `user_id`, `recipe_id`, `created_at`
- **CookingLog**: `id`, `household_id`, `user_id`, `recipe_id`, `servings`, `cooked_at`

## 8. 外部インターフェース要件

外部依存サービスとその用途を以下に示す。具体的なプロバイダは設計フェーズで確定する。

- **LLM**: レシピ生成・レシート補正に利用（OpenAI / Anthropic 等）
- **画像生成API**: 完成イメージのAI生成に利用（DALL·E / Stable Diffusion 等）
- **OCR**: レシート文字認識に利用（Google Cloud Vision / Tesseract 等）
- **メール送信**: 招待・パスワードリセット通知に利用

## 9. 制約・前提条件

- 個人利用規模を想定し、初期構成はサーバコストを抑える方針とする
- 機密情報（APIキー等）は環境変数で管理し、リポジトリにはコミットしない
- 文字コードはUTF-8、改行コードはLFを使用する
- UI・ドキュメントの基本言語は日本語とする

## 10. 運用要件

- DBの日次バックアップを取得する
- 主要APIのエラーレート・レイテンシを監視する
- LLM・画像生成APIの利用コストに上限アラートを設定する
- 利用規約・プライバシーポリシーを整備し、レシート画像・食材データの取り扱いを明示する

## 11. 開発スコープとマイルストーン

MoSCoW を踏まえ、段階的に開発を進める。

### 11.1 MVP（最初のリリース）

1. アカウント認証（FR-01）
2. 世帯作成・招待（FR-02）
3. 食材手動登録・編集・削除（FR-11, FR-12, FR-13）
4. レシピ提案・詳細表示（FR-20, FR-21, FR-23, FR-25）
5. 作った減算（FR-30, FR-31）
6. お気に入り（FR-40, FR-41）
7. アレルギー食材登録と除外（FR-50, FR-52）

### 11.2 Phase 2

1. レシート画像アップロード（FR-10）
2. AI完成イメージ画像生成（FR-22）
3. 調理履歴記録（FR-32）
4. 常備食材・調味料管理（FR-14）
5. 提案条件指定（FR-24）
6. 苦手食材の優先度反映（FR-51）

### 11.3 Phase 3 以降（将来案）

1. 買い物リスト自動生成
2. 献立カレンダー
3. 栄養バランス可視化
4. 食材消費トレンド分析

## 12. 残課題・今後の検討事項

- LLM出力のばらつき対策（プロンプト設計、評価基準）
- レシートOCRの精度（店舗ごとのフォーマット差異）
- 「ひとつまみ」「少々」など曖昧分量と在庫数量の突合せ方
- カロリー・糖質など食事制限への将来拡張
- AI生成画像の著作権・コスト
- PWAオフライン動作のキャッシュ範囲
